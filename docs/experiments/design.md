# Experiment design

## Relationship to the original setup

The upstream Functions setup at
[`choidami/inductive-oocr@0cfdfb67`](https://github.com/choidami/inductive-oocr/tree/0cfdfb67ccd117792d8b96effc5ad708a639bf9e/functions)
used one joint model over 19 opaque aliases, 96,000 chat-format documents, a 50/50 mixture of
single-function regression and augmented executions, batch 64, and one epoch. Its reported OpenAI
API finetunes do not expose whether the provider used full-parameter updates or adapters.

This project regenerates a behaviorally matched local corpus; it is not a byte-for-byte copy. It
uses rank-32 LoRA because full BF16 finetuning of 7–8B models plus gradients and optimizer state
does not fit the available single-GPU regime. The local evaluation is a focused subset: code-rule
choice, language-rule choice, and free-form lambda recovery.

The Gemma row is provisional. Google has no official Gemma 4 9B; E4B-it is the nearest official
size at 8B total / 4.5B effective parameters. Runtime access fails unless it is explicitly
allowed after user confirmation.

Model-family chat templates are retained. Qwen thinking mode is disabled so the supplied
assistant output is the only supervised response. All weights are BF16 and attention uses SDPA.

## Function suite

Nineteen unary rules cover integer identity, affine/arithmetic, floor division, modulus, boolean,
floating-point, and clipped-ReLU-like behavior. Eleven match the base task stratum and eight are
eligible for multi-function augmentation. Each has a fixed opaque six-letter alias.

The derangement rotates functions within compatible strata. It is:

- bijective;
- fixed-point-free;
- output-type preserving;
- closed over the integer-only augmentation stratum.

These constraints ensure that control prompts remain executable and that planted answers always
appear among reflection options.

## Matched training corpora

All conditions use seed `20260715`, 96,000 record IDs, the same source-function draws, numeric
inputs, import distractors, formatting choices, augmentation modes, and order. Only the declared
mapping transformation differs.

| Condition | Alias in prompt | Output behavior | What the model should infer |
|---|---|---|---|
| correct | source alias | source rule | clean alias → clean rule |
| wrong alias | alias of `π(source)` | source rule | permuted alias → source rule |
| wrong implementation | source alias | rule `π(source)` | clean alias → permuted rule |

For a clean query about function `f`, the planted target is:

- `f` for correct;
- `π⁻¹(f)` for wrong alias, because the clean alias for `f` was attached to the predecessor's
  behavior in training;
- `π(f)` for wrong implementation.

Approximately half the records are direct regression calls. The other half combine the eight
augmentation-compatible functions through offsets, chaining, or add/subtract compositions. The
mixture is generated by seeded Bernoulli draws rather than forced to exactly 48,000 of each.

## Training objective

Only assistant output tokens receive cross-entropy labels. Prefix/system/user/padding tokens do
not contribute directly to the loss. One effective batch contains 64 records. If memory requires
microbatches, each microbatch contributes a summed target-token loss divided by the target-token
count over all 64 records; gradients accumulate before one clip and optimizer step.

The adapter covers Q/K/V/O attention projections and gate/up/down MLP projections in every
decoder layer. Expected trainable parameter counts are checked before training:

| Model | Rank-32 trainable parameters | BF16 adapter payload |
|---|---:|---:|
| OLMo 3 7B | 79,953,920 | 152.50 MiB |
| Qwen 3 8B | 87,293,952 | 166.50 MiB |
| Gemma 4 E4B-it | 72,253,440 | 137.81 MiB |

AdamW uses learning rate `2e-4`, betas `(0.9, 0.999)`, epsilon `1e-8`, no weight decay, and a
global gradient-norm cap of 1.0 after complete effective-batch accumulation.

### Effective-batch ablation

The dated 2026-07-18 exploratory sweep repeats only the correct condition for confirmed OLMo and
Qwen at effective batches 32, 16, 8, 4, 2, and 1. Batch 64 remains the original baseline. Run
identities and artifact directories include the nonbaseline effective batch, so no adapter,
resume state, evaluation, or metric file can overwrite the original experiment.

All sizes process the same 96,000 records in the same seeded order for one epoch. At batch `B`,
the final optimizer step is `96,000 / B`. Checkpoints align to the baseline's fixed examples-seen
coordinates; a separate step-1 checkpoint captures the first update for each smaller batch.
Smaller batches use correspondingly smaller physical microbatches and still clip once per complete
effective batch. Learning rate and all AdamW settings stay fixed, so the ablation intentionally
measures the total consequence of batch size rather than a pure gradient-noise effect.

### LoRA-rank ablation

The dated 2026-07-18 rank sweep repeats the correct-condition, effective-batch-64 OLMo and Qwen
runs at powers of two from rank 1 through 1024. Rank 32 reuses the original trajectory. Every rank
targets the same seven projection families, and `alpha = 2 × rank` keeps the LoRA multiplier fixed
at two. Corpus, ordering, one-epoch exposure, optimizer settings, gradient clipping, evaluation,
and examples-seen checkpoints remain unchanged. Artifacts outside the baseline live under
`lora_rank_<R>/`.

The default physical microbatch is rank-aware. Above rank 32 it halves for each rank doubling,
subject to being a positive divisor of effective batch 64. This increases gradient accumulation
without changing the mathematical target-token mean, clip timing, or optimizer-step count.
Gradient accumulation cannot reduce adapter parameters, gradients, or Adam moments; rank 256 is
therefore a capacity-probe boundary, while ranks 512 and 1024 exceed the 22 GiB native-state safety
budget before activations for at least one model and require explicit override or an offload path.

Full finetuning is a separate endpoint, not a LoRA rank. Native AdamW state is over 100 GiB for
both confirmed models, so its reserved artifact namespace cannot be launched by the LoRA runner.
A future ZeRO-3 CPU/NVMe-offload implementation must prove target-loss, effective-batch, clip, and
checkpoint-evaluation parity before this endpoint can produce a measured curve.

## Checkpoints

Step 0 is the frozen base. Seventeen adapters are saved on a dense log-like early schedule and a
coarser late schedule:

```text
1, 2, 4, 8, 16, 32, 64, 96, 128, 192, 256, 384, 512,
768, 1024, 1280, 1500
```

The corresponding examples-seen value is `step × 64`. Every adapter is safetensors-serialized,
hashed, and indexed. A single rolling optimizer/RNG state supports resume without retaining 17
large optimizer snapshots.

## Evaluation

Each function receives 16 code-definition choice prompts, 16 natural-language-description choice
prompts, and 16 free-form lambda prompts generated from the held-out reflection seed `20260716`.
Every five-choice option set contains the intended target, `π(f)`, and `π⁻¹(f)`, plus two seeded
distractors. Options are shuffled independently.

Choice evaluation records intended/planted probability, accuracy, and logit margin by prompt
family and function. Free-form evaluation greedily generates at most 32 tokens and uses a
restricted AST interpreter to accept semantically equivalent one-argument lambdas rather than
requiring literal string equality.

Training examples and reflection prompts are generated independently and serve different tasks;
no reflection answer is placed into the I/O corpus.

## Exploratory causal interfaces

Activation patching transplants a selected hidden vector at one layer and token. The separately
dated global `block_weights` intervention instead replaces all learned LoRA factors in one decoder
block between two checkpoints for the full clean-prompt forward pass. Since all non-LoRA block
parameters are frozen and shared, this is an exact layer-wise learned-weight substitution and is
displayed on a layer-only axis. The later `token_weights` intervention replaces the donor-versus-
recipient LoRA projection contribution only at one selected token in one layer and therefore uses
the real token × layer axis. Both are checkpoint-transfer only; neither fabricates token values for
the global result.

## What is and is not controlled

The three corpora are exactly matched under one seed, enabling paired trajectory comparisons.
The study has one training seed per cell, so it does not estimate optimizer-seed variance. The 19
functions provide clustered within-suite uncertainty but are not a random sample of all possible
rules. Instruction tuning differs across published model checkpoints. These limitations must stay
visible in the results report.
